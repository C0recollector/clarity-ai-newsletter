Model Routing: Stop Asking for One Best Model

The model market is splitting into specialized lanes: coding, realtime voice, open and efficient models, image/media, and science-oriented systems.

01

Model Routing Map

Codingrepair, refactor, tests
Realtimevoice, support, translation
Mediaimage, video, creative
Policy Router
cost + privacy + latency + quality
Frontierplanning and hard reasoning
Efficientclassification and formatting

Architecture move

Create a routing layer that chooses model class by task, sensitivity, latency, and price.

Eval move

Keep a small regression set for each workflow instead of relying on public benchmarks.

Cost move

Use frontier models for planning and hard reasoning; use cheaper models for extraction, formatting, and classification.

Technical takeaway: the durable asset is the routing/evaluation harness, not whichever model wins the week.

Routing Checklist

  • Inventory the top 10 AI workflows by volume and business risk.
  • Tag each workflow with latency, privacy, reasoning depth, and quality requirements.
  • Run two vendors and one lower-cost model against the same task set.

Agent Architecture: Workflow First, Memory Later

Agents are becoming execution surfaces across files, folders, tools, browsers, documents, code, and workflows.

02

Agent Workflow Architecture

Job Contractgoal, inputs, done criteria
Tool Boundaryfiles, apps, APIs, browser
Artifact Traildrafts, diffs, sources, logs
Review Gateapprove, publish, archive
Design unit

Define the job as input -> allowed actions -> output artifact -> review gate.

State model

Use explicit files, tickets, docs, and run logs as durable state before relying on generic memory.

Permission model

Separate read, draft, edit, publish, spend, and external-send permissions.

Technical takeaway: agents need an operating procedure more than they need a personality.
First Pilot

Newsletter production: gather sources, build review, approve themes, generate pages, publish.

Failure Mode

Vague goals create long-running loops. Strong artifacts keep the work inspectable.

Operating Notes

  • Write the workflow contract in one page.
  • Add a review gate before public publishing.
  • Record sources, generated outputs, and final URL in the issue folder.

Platform Integrations: AI Is Moving to the Point of Work

AI is being embedded where users already work, search, design, and decide.

03
UX move

Map where users currently ask, search, copy/paste, summarize, or hand work across systems.

Integration move

Prioritize extensions, APIs, and embedded actions inside existing work surfaces.

Measurement move

Track answer quality, task completion, conversion, and handoff reduction.

Technical takeaway: AI adoption is easier when the assistant lives where the work already happens.

Integration Scan

  • Review search journeys that may become AI answer journeys.
  • Identify internal workflows where browser or document agents can remove handoffs.
  • Prepare attribution changes if AI answers reduce normal web traffic.

Compute Ops: Capacity Is Now an Architecture Constraint

Compute, memory, power, cooling, data centers, and strategic access to capacity are becoming business constraints.

04
Reliability move

Define fallback behavior for rate limits, vendor degradation, or budget caps.

Data move

Document where prompts, files, logs, and derived outputs can be stored.

FinOps move

Report cost per workflow, not just total AI subscription spend.

Technical takeaway: production AI needs the same discipline as cloud operations: capacity, observability, incident plans, and cost controls.

Production Controls

  • Set max tokens, max runtime, and max spend per workflow.
  • Cache repeatable retrieval and generated reference material.
  • Keep a vendor capacity and data-residency questionnaire.
